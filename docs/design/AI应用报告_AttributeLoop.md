# 《AI应用报告》— AttributeLoop

> 本报告据实记录 AttributeLoop（Godot 4 折纸题材 Roguelike）开发过程中 AI 工具的使用情况。

---

# AI应用场景总览

**一句话概括：**

> AttributeLoop 是一款由程序员独立开发的折纸题材 Godot 4 Roguelike。开发者完全没有美术基础，也非叙事/文案专业。本作品利用 AI 解决了四类问题：① 全量美术资源的程序化生成；② 背景音乐的生成；③ 世界观与文案撰写；④ 游戏系统的端到端开发、自动化测试与工程文档。**如果没有 AI，这款游戏根本无法由一个人独立完成**——不是效率高低的问题，而是从 0 到 1 的可行性问题。

**工具清单：**

> - **AIArt** —— AI 图像生成，支持文生图 / 图生图 / 风格参考 / 去背景 / 放大，承担全部美术资源
> - **Suno** —— AI 音乐生成，由 Claude Code 对接生成背景音乐
> - **Claude Code** —— 作为编码 Agent 框架承担代码开发、调试、测试、文档与世界观撰写，并对接 AIArt / Suno 完成美术与音乐资源的调度。编辑器统一使用 Claude Code，但底层模型按任务特点多模型混用：GLM 5.2、GPT 5.5、Sonnet 4.6，三者各有所长、按需切换
> - **Godot MCP** —— 让 Claude Code 直接操作 Godot 编辑器的插件，支持创建节点/场景/脚本、运行主场景、截图验证，是端到端开发闭环的关键

**AI 在四个环节的作用：**

> **① 美术资源生成 —— AIArt**
> 开发者是程序员，完全没有美术基础，无法手绘任何一张图。游戏需要 30+ 组件图标、5 类敌人精灵及动画帧、玩家精灵、10+ UI 面板、8 张叙事背景。这些资源全部由 AIArt 生成，是游戏能够"有画面"的唯一可行路径。
>
> **② 音乐生成 —— Suno**
> 开发者同样没有音乐制作能力。游戏的背景音乐由 Claude Code 直接对接 Suno 生成：Claude Code 根据折纸题材的氛围需求描述曲风与情绪、调度 Suno 出片并筛选，最终产出的音频直接接入游戏的背景音乐系统。全程无需任何乐理知识或宿主软件操作，程序员也能为游戏配乐。
>
> **③ 世界观与文案撰写 —— Claude Code**
> 开发者非叙事/文案专业，独立构建一套折纸题材的世界观——折纸师的未竟之愿、五种折叠型敌人、机制与设定的映射，并撰写各阶段叙事背景文案、剧情对白与 UI 内的说明文字。Claude Code 在此承担世界观共创与文案打磨的角色——没有 AI，能否写出一段可用的设定都不一定。
>
> **④ 端到端开发与测试 —— Claude Code + Godot MCP**
> 这是本项目把 AI 利用得最彻底的环节。Claude Code 通过 Godot MCP 直接驱动编辑器，形成「写代码 → 跑 headless 单元测试 → 运行主场景截图可视验证 → 写模块文档」的自测闭环。当前自动化测试 362/362 通过、1171 项断言全绿，这套流程让 AI 不只是"写代码的工具"，而是"能自主验证、自主纠错的开发伙伴"。

**效率提升：**

> 本项目为独立开发，截至当前已完成 Phase 1–5 的核心内容——可行走的世界 → 属性规则战斗 → 永久地块 → 商店与服务系统 → 裁决圈 Boss 战，游戏可从开局完整通关，含 6 阶段叙事过渡、强制首启教程、角色面板与数值平衡；全套美术资源与 BGM 就位；自动化测试 362/362 通过、1171 项断言；16 份模块文档沉淀。整体历时约 7 周（4/29–6/19），657 次提交。
>
> 若无 AI：美术外包 + 系统代码独立手写 + 文档与世界观独立撰写，保守估计 4–6 个月；而考虑到开发者无美术基础、非叙事专业，部分环节即使给时间也无法由个人独立完成。因此 AI 带来的不仅是约 60–70% 的时间压缩，更是**让"单人完成一款有美术、有剧情、有测试、可通关的 Roguelike"从不可能变为可能**。

# 作品关键AI应用阐述

## 关键应用一：全量美术资源生成（AIArt + 自研 art-production 自动化流程）

**痛点：** 开发者是程序员，完全没有美术基础，无法手绘任何一张图。而游戏需要一套风格统一的折纸题材美术：30+ 组件图标、5 类敌人精灵（每类含待机与激活两套动画帧序列）、玩家精灵与行走动画、10+ UI 面板与按钮、8 张叙事背景，合计上百件资源。这是游戏能否"有画面"的根本前提。

**主要工具：** AIArt，配合自研的 art-production 技能（skill）由 Claude Code 在终端全自动调度

**说明详细过程：**

1. **核心做法——自研 skill 把全流程在终端自动化**：本项目没有采用"在 AIArt 网页上逐张手点"的常规用法，而是写了一个 art-production 技能，让 Claude Code 在终端里把整条生产流水线全自动串起来：读取资源清单 → 拼装提示词 → 调用 AIArt 生成 → 把变体以链接形式列在终端供挑选 → 去背景 → 下载保存到对应目录 → 在清单里打勾 → git 提交。开发者全程不打开浏览器、不在网页上操作，只在关键节点做选择，其余由 skill 自动完成。
2. **Style Bible 先行**：skill 首次运行会先建立一份可复用的风格圣经——针对背景、图标、UI 三类资源各写一套正向提示词（背景用《你的名字。》风格手绘动画、蓝灰冷调配暖金光晕；图标用小王子灵感柔和手绘、深底单符号；UI 用泛黄羊皮纸配手绘墨边），再配一份负向提示词统一排除照片级写实、3D 渲染、像素、真人、水印等。风格圣经一旦定稿，后续所有资源共享，保证全项目风格统一。
3. **资源清单驱动的批量生产**：以一份资源清单作为单一真实来源，skill 自动统计总数、已完成、待办，并定位下一个待办项，按资源类型自动走对应路径，无需人工逐张安排。
4. **锚定图机制保持跨批次一致**：先评审出满意的背景与地面色调图作为锚定，之后所有同类生成调用都以低权重把锚定图作为参考传入，让新图始终向已认可的风格靠拢，控制 AI 生图天然的风格漂移。
5. **两条自动化路径**：
   - 静态资源路径（图标、UI、地块、背景）：拼提示词 → 生成多个变体 → 终端列链接供挑选 → 按需去背景 → 下载保存 → 清单打勾 → 提交。
   - 动画精灵路径（玩家、敌人）：先锁定角色正面参考图，再以参考图驱动生成动画视频，skill 自动调用 ffmpeg 按指定帧率抽帧、批量去背景（多帧并行流水线）、下载落地，并更新游戏代码加载正确的帧数与帧率。
6. **人机协作的迭代节点**：每个资源生成多个变体后，skill 把变体链接贴在终端，开发者选号或给反馈，skill 据此自动调整提示词重生成，迭代次数无上限，直到满意才保存。AI 负责生产与机械操作，人只做审美判断。

**产出结果：** 全套美术资源就位，覆盖图标、敌人与玩家精灵及动画、UI 面板与按钮、叙事背景等全部品类，风格统一；整个生产过程由 skill 在终端自动推进，清单可追踪、可复跑。

**效率提升：** 同等规模美术若外包，按上百件资源估算，沟通与返工周期约 6–8 周（约 240–320 工时）；若用 AIArt 但逐张手动操作，上百件资源的人工点击、下载、归类也是沉重的体力负担。借助 art-production skill 的终端自动化流水线，压缩到约 1–2 周（约 40–80 工时，含人工审美筛选），**提速约 4–6 倍**，且对一个无美术基础的独立开发者而言，这是唯一可行的路径。

## 关键应用二：背景音乐生成（Suno）

**痛点：** 开发者同样没有音乐制作能力，不掌握乐理，也不会使用任何数字音频宿主软件，但游戏需要契合折纸题材氛围的背景音乐。

**主要工具：** Suno 官方 API，由 Claude Code 编写脚本对接调用

**说明详细过程：**

1. **Claude Code 通过 API 脚本对接 Suno 官方**：开发者提供 Suno 官方 API 密钥，Claude Code 据此编写并运行调用脚本，通过 API 直接对接 Suno 官方服务，无需打开 Suno 网页、无需人工在界面上操作。整条调用链由脚本自动完成：构造请求 → 调用官方接口 → 拉取生成结果 → 下载音频。
2. **由 Claude Code 把游戏调性翻译成曲风需求**：不写乐谱、不碰宿主，而是用自然语言描述目标——折纸题材、安静略带神秘、适合长时间探索的舒缓循环氛围。Claude Code 根据游戏整体调性把这些需求组织成 Suno 能理解的生成提示。
3. **批量生成多首候选**：一次调用生成多首候选曲目，覆盖不同的情绪倾向与编曲方向，形成可挑选的素材池，全部下载到本地。
4. **试听筛选与接入**：逐首试听，按是否契合氛围、是否适合循环播放筛选，选定曲目接入游戏的背景音乐系统。
5. **工程演进与按需精简**：最初按"探索 + 战斗"双场景设计，实现了多首 BGM 配合三态机（探索 / 战斗 / 静音）与 1 秒淡入淡出切换的完整系统并跑通。实际试玩评估后认为单曲循环 BGM 已能覆盖体验需要，于是精简为单首循环，删除多余曲目与切换逻辑。这一步体现的不是 AI 的能力，而是借助 AI 快速搭建→试玩→精简的迭代节奏：先跑通完整方案，再据实删到刚好够用。

**产出结果：** 由 Claude Code 调用 Suno 生成多首 BGM 候选，筛选后接入游戏作为循环背景音乐，音乐氛围与折纸题材美术风格统一；完整的多曲切换系统曾跑通并按需精简。

**效率提升：** 若走传统路径，独立开发者要么外包配乐（沟通与返工约 2–4 周），要么自学宿主与乐理（数周到数月门槛）；借助 Claude Code 对接 Suno 的全自动调用链，把配乐压缩到约 2–4 小时（含多首候选生成与筛选），**提速约 30 倍以上**，且全程无需任何音乐专业操作。

## 关键应用三：世界观与文案撰写（Claude Code）

**痛点：** 开发者非叙事/文案专业，却要独立构建折纸题材世界观、贯穿全流程的多阶段叙事弧、游戏结局文案及各 UI 说明文字。

**主要工具：** Claude Code

**说明详细过程：**

1. **世界观共创**：与 Claude Code 搭两层叙事底座——底层是折纸世界观（折纸师未竟之愿、机制即折叠模式），其上是一条共享梦境的情感叙事：你与一个未知的人在亭子通过石子与刻字隔空交流，至裁决前夜离别。世界观在项目最初先行确立，作为后续整体美术风格与音乐氛围的统辖依据，美术的蓝灰冷调 + 暖金光晕、音乐的氛围基调都由这个世界观推导而来。再由 Claude Code 配胜负结局文案与角色面板、商店、教程等各 UI 说明文字，保证措辞与世界观调性一致。
2. **6 阶段叙事弧**：最终落地为 6 段递进的故事（觉醒 → 涌动 → 侵蚀 → 失衡 → 裁决前夜 → 裁决前夜 Boss），以"石子"为承上启下的伏笔贯穿——第 2 阶段摆出问号、第 5 阶段收束于"那个石子摆成的问号，早已经有了答案"，前后呼应。
3. **polish-only 协作纪律**：定稿后约定 Claude Code 只做措辞润色，不删节意象、节拍、连接线与伏笔。这条纪律让人机协作可控——AI 负责提升表达力，叙事骨架的决定权始终在人。
4. **守护测试与文档沉淀**：为防止润色误删承上启下的关键句，专门写单元测试断言石子伏笔等句子的存在，润色一旦删掉关键句测试立即失败，用工程手段守住叙事完整性。全过程同步沉淀为世界观文档、叙事设计文档与模块文档，文案不止落在代码里，也有设计文档解释 motif 走向，有据可追溯。

**产出结果：** 完整世界观、6 阶段叙事弧、胜负结局与 UI 文案；配套设计文档与模块文档沉淀，关键伏笔由测试守护。

**效率提升：** 非叙事专业者独立撰写一套完整世界观与多阶段剧情几乎不现实，若外包约需 2–3 周；借助 Claude Code 协助共创、压缩与打磨，约 2–3 天即产出可用文案，**提速约 5–7 倍**。

## 关键应用四：端到端开发与自测（Claude Code + Godot MCP）

**痛点：** 独立开发一套含属性规则、循环轨道、战斗、HUD、教程、商店、数值平衡的 Godot 4 Roguelike，系统复杂度高，单人难以同时兼顾架构、实现、测试与文档。本应用是项目里把 AI 利用得最彻底的环节，其核心价值不在"AI 能写代码"，而在 **AI 能自主验证、自主纠错**。

**主要工具：** Claude Code，配合 superpowers 技能体系（设计→执行流程 + 测试驱动）与 Godot MCP 插件

**说明详细过程：**

1. **第一套 skill——superpowers 设计→执行流程**：每个功能先用 brainstorming 对齐需求与设计，复杂功能再用 writing-plans 产出设计文档与分步实施计划，最后用 executing-plans 按计划逐步执行、每步设 review checkpoint；计划中的独立任务可由 subagent-driven-development 分发给子代理并行实现，加快推进。这套流程把模糊需求变成可追踪的分步执行，而非一上来就写代码。项目累计沉淀了数十份设计文档与实施计划配对，每个功能都走得清、查得到。
2. **第二套 skill——测试流程嵌入执行**：test-driven-development 让每个 task 走 RED-GREEN-REFACTOR——先写测试看到失败（RED），再实现到测试通过（GREEN），再重构。测试不是事后补，而是嵌进执行的每一步，作为进度的判据。两套 skill 的关系是：第一套规定"怎么从需求走到完成"，第二套保证"每一步完成都有测试兜底"。
3. **三层自测协议 + 两类测试**：每个 task 完成后跑"语法检查 → headless 单元测试 → 运行主场景截图可视验证"三层，全绿才算完，任何一层失败 AI 自行定位、修复、重跑。测试分两类——LOG 测试在 headless 下验证逻辑与状态转移，截图测试运行后由 Claude 读图按视觉断言判断 UI 是否正确。当前 362/362 通过、1171 项断言全绿。这套流程让 AI 从"写代码的工具"升级为"能自主验证、自主纠错的开发伙伴"。
4. **Godot MCP 直驱 + 文档沉淀 + 提交纪律**：AI 通过 MCP 自己在编辑器里创建节点、场景、脚本，运行主场景并截图，不靠人工点选操作，真正端到端自动化；每完成一个模块即写一份模块文档，累计 16 份，记录职责、关键类、执行流与依赖，代码与文档始终同步；每个 task 完成即 git 提交，全程 657 次提交，每步可回溯。按 5 阶段 Roadmap 推进，当前已完成 Phase 1–5 核心内容，游戏可从开局完整通关至裁决圈 Boss 战。

**产出结果：**

- **代码规模**：约 5000 行 GDScript 生产代码（51 个文件）、64 个场景文件
- **测试规模**：约 2600 行测试代码（35 个文件）、362 个测试用例全部通过、1171 项断言全绿；测试代码量约为生产代码的一半。这种覆盖密度在常规项目组几乎不可能达成——人工写测试的边际成本会迅速压垮进度，而 AI 按 TDD 流程逐任务先写测试再实现，把测试当作开发的主线而非负担，才让覆盖率达到这个水平
- **文档沉淀**：16 份模块文档 + 数十份设计文档与实施计划配对
- **提交记录**：657 次 git 提交（4/29–6/19，约 7 周），其中新功能（feat）228 次、调优类（修复 / 重构 / 性能 / 数值平衡）246 次——调优工作量与新功能建设相当，印证"前期 3–4 周搭功能、其余持续调优打磨"的开发节奏
- **里程碑**：Phase 1–5 核心内容完成，游戏可从开局完整通关至裁决圈 Boss 战

**效率提升：** 核心功能开发约 3–4 周（无 AI 估算 8–12 周），**提速约 2–3 倍**；含持续调优与打磨全周期约 7 周。AI 不只是更快，而是把"高测试覆盖 + 完整文档"这种正常项目组难以兼顾的工程纪律落到了实处——657 次提交里调优类占了近一半，正是 AI 让"反复打磨"这种人力成本极高的环节得以持续进行。

# AI应用的更多思考

- **AI 美术生成——风格统一与批量自动化仍是瓶颈。** 虽已用 art-production 流程自动补全关键词、以锚定图控制风格，但整体风格仍不够统一；图片一多，总会有部分素材发生偏移。有时无论怎么调整提示词，AI 都生成不出想要的效果，批量跑也难——量一大，风格偏移会累积。这是后续最需要继续投入的方向：把视觉方案进一步嵌入图片生成的自动化流程，让风格控制在批量生产中更稳定，而不是靠人工逐张筛选兜底。
- **代码自动化自测——已达到可上生产线的状态。** 这是本次 AI 表现最好的环节。常见的做法是：设定好模块任务后让 Claude Code 跑一整晚，只要带上了自测流程，第二天游戏启动后基本零报错、稳定运行。从"写完代码"到"自测验证通过"形成闭环后，AI 的产出已经达到可以放心交给生产线的可靠度。
- **视觉测试——LLM 的短板，需要引入视觉模型。** 视觉测试这块 LLM 表现还不够完善，尤其在调整交互界面时经常不如人意。有时花大力气跟 AI 讨论界面调整，效率还不如直接人工上手微调。根本原因是 LLM 本质面向编程与写作，对像素级视觉判断力有限。后续考虑接入专门的视觉模型来承担 UI 视觉验证，而非局限于 LLM 的文字判断，让"截图自测"这一层真正闭环。
